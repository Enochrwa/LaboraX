## 2. Request Lifecycle (Example: Submit Interpretation)

1. `POST /api/v1/interpretations` with `{case_id, free_text}` + JWT
2. `deps.get_current_user` validates JWT, resolves `student` role
3. Route handler calls `answer_evaluator.evaluate(case_id, free_text, db)`
4. `answer_evaluator`:
   - Loads case's expected-findings reference from DB (`case.expected_findings`)
   - Runs spaCy preprocessing + sentence-transformer embedding on student text
   - Computes similarity against each expected finding embedding
   - Extracts confirmed / missing / incorrect findings via threshold + rule checks
   - Computes composite score (0–100)
5. Result persisted to `interpretation_results` table; `student_topic_mastery` updated (async, via ARQ task if computation is non-trivial)
6. Response returns score, confirmed/missing findings, and tutor explanation text generated by `case_generator`'s domain rule templates (not a free-form LLM call)

```

results (id, case_id FK, test_id FK, result_payload JSONB, generated_at)

interpretation_results (id, case_id FK, student_id FK, submitted_text TEXT,
                          score NUMERIC, confirmed_findings JSONB, missing_findings JSONB,
                          incorrect_findings JSONB, tutor_feedback TEXT, evaluated_at)

student_topic_mastery (id, student_id FK, topic, mastery_score NUMERIC, updated_at)

case_assignments (id, lecturer_id FK, case_id FK, assigned_to_group, due_at, created_at)

recommendations (id, student_id FK, recommended_case_id FK, reason, created_at)
```

Indexes: `cases(disease_id)`, `interpretation_results(student_id, case_id)`, `student_topic_mastery(student_id, topic)` unique.

## 5. Answer Evaluation Algorithm (Detail)

1. Normalize student text (lowercase, lemmatize via spaCy)
2. Split into candidate finding-statements (sentence segmentation)
3. Embed each candidate statement + each expected-finding statement using a small sentence-transformer model
4. Compute cosine similarity matrix; match each expected finding to its best-scoring candidate statement
5. Apply similarity threshold (e.g., ≥0.75 = confirmed, 0.5–0.75 = partial, <0.5 = missing)
6. Flag statements in the student text that don't match any expected finding and contradict known case findings as "incorrect"
7. Composite score = weighted sum (confirmed=full weight, partial=half weight) normalized to 100, minus a small penalty per incorrect statement (floor at 0)

## 8. Testing Strategy

- **Unit:** each service (`case_generator`, `answer_evaluator`, `result_predictor`, `recommender`) tested in isolation with fixed seeds/fixtures
- **Integration:** FastAPI `TestClient`/`AsyncClient` end-to-end route tests against a test Postgres schema
- **ML sanity checks:** golden-case regression tests (known case → known expected evaluation output within tolerance)
- **CI gates:** ruff, mypy, pytest with coverage threshold, ESLint/Prettier + tsc for frontend

## 9. Configuration & Secrets

- All secrets via environment variables (`.env` locally, platform secret manager in deployment)
- No secrets committed to the repository — see `.gitignore`
- `core/config.py` centralizes and validates all environment-derived settings via Pydantic `BaseSettings`
